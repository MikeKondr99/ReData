# AGENTS.md

## Назначение
- Этот файл читается первым при работе с репозиторием и задает локальные правила для ИИ.
- После `AGENTS.md` ИИ должен прочитать `docs/ai.md` (если файл существует).
- Разрешено самостоятельно читать связанные `*.md` файлы для контекста (включая `docs/*` и другие папки), даже без явного запроса пользователя.
- Новые документы можно добавлять в `docs/*.md`, но не обязательно выносить все туда. Существующие файлы в `docs/` можно редактировать, если изменения в коде делают их неактуальными.

## Что умеет проект (в общем виде)
- Реализует собственный язык запросов и движок выполнения выражений.
- Поддерживает построение запросов, компиляцию и выполнение по различным источникам данных (в т.ч. SQL‑бэкенды).
- Имеет инструменты импорта/экспорта данных (CSV/Excel).
- Содержит демонстрационное веб‑приложение и фронтенд на Angular.

## Solution и проекты
- `src/ReData.Query.Lang` — парсер и грамматика языка (ANTLR), построение выражений.
- `src/ReData.Query.Core` — ядро запросов: выражения, резолверы, query builder.
- `src/ReData.Query.Common` — общие типы/ошибки/кэш для модуля запросов.
- `src/ReData.Query` — интеграционный слой: компиляция, источники/раннеры запросов, коннекторы БД.
- `src/ReData.DataIO` — импорт/экспорт данных и форматы (CSV/Excel).
- `src/ReData.Common` — общие утилиты и расширения.
- `src/Pattern` — вспомогательные шаблоны/типы.
- `src/ReData.DemoApp` — демонстрационное серверное приложение (эндпоинты, сервисы, трансформации).
- `src/ReData.Angular` — фронтенд/SPA на Angular.
- `src/*Tests` — тестовые проекты (язык запросов, имплементации, demo app).
- `src/FileExporterBenchmarks` — бенчмарки экспорта.

## Команды (типовые для .NET)
- `dotnet restore`
- `dotnet build`
- `dotnet test`
- `dotnet run --project src/ReData.DemoApp`
- Docker-команды см. `docs/commands.md`

## Сборка при блокировке файлов
- Если сборка/тесты падают из-за блокировки `obj/bin` запущенным процессом (`.NET Host`, `ReData.DemoApp` и т.п.), агент должен завершить мешающий процесс и повторить команду.
- Не использовать для этого обходные пути со сборкой в нестандартные директории (`.tmp`, `buildcheck`, кастомные `BaseIntermediateOutputPath/IntermediateOutputPath`).

## Кодстайл
- Следовать `.editorconfig` и допускать минимальное число предупреждений.
- Не использовать вложенные/цепочные тернарные операторы (пример из `src/ReData.Query/Functions/Library/ReflectionFunctions.cs`): при нескольких ветках использовать `switch`/`switch expression` или `if/else`.
- Все изменения, вносимые агентом (и результаты ревью, если затрагивают код), должны быть достаточно полно покрыты тестами.

## Работа с багфиксами
- Фикс бага должен включать не только изменение кода, но и проверку пользовательского сценария: где падает, какой статус/ответ должен быть после фикса.
- Если возможно, вместо необработанного `500` возвращать контролируемый ответ (`4xx`/`5xx`) с понятной причиной.
- Для регресс-тестов по багам обязательно оставлять трассировку к источнику:
  - добавить атрибут `[Trait("Issue", "<url-на-issue>")]`;
  - добавить короткий комментарий в тесте вида `Regression test for GH-...`.

## Стиль коммитов
- Использовать Conventional Commits.
- Разрешенные типы: `feat`, `fix`, `refactor`.
- Формат: `<type>(<scope>): <subject>`.
- Разрешенные scope: `functions`, `qb`, `api`, `gui`, `test`.
- Для остальных изменений использовать scope `core`.

## Обновление документации (обязательно)
- При любых изменениях кода агент обязан проверить актуальность документации и обновить затронутые `*.md` файлы в том же изменении.
