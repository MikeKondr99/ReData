# BUG-006: Нет валидации дублирующихся alias в `select` при сохранении/обновлении набора

## Описание что не так
- Трансформация `select` с двумя `items` с одинаковым `field` (alias) сохраняется через `POST /api/datasets` с `201`.
- Аналогично проблема воспроизводится через `PUT /api/datasets/{id}`: невалидная конфигурация сохраняется с `200`.
- Такой набор потом не исполняется: `GET /api/datasets/{id}/export` возвращает `500`.

## Шаги воспроизведения (вариант `POST`)
- Создать набор с `select`:
  - `{"field":"x","expression":"id"}`
  - `{"field":"x","expression":"age"}`
- Убедиться, что создание вернуло `201`.
- Запросить экспорт набора.
- Получить `500`.

## Шаги воспроизведения (вариант `PUT`)
- Создать валидный набор.
- Выполнить `PUT /api/datasets/{id}` с `select`:
  - `{"field":"x","expression":"id"}`
  - `{"field":"x","expression":"age"}`
- Получить `200`.
- Проверить `GET /api/datasets/{id}`: дубли alias сохранены.
- Запросить экспорт набора.
- Получить `500`.

## Ожидаемое поведение
- Дубли alias должны валидироваться на этапе сохранения набора с понятной ошибкой `400`.
- Невалидная конфигурация не должна попадать в исполняемый набор.

## Возвращенный ответ API
- При `POST`: `Status 201`.
- При `PUT`: `Status 200`.
- При экспорте после сохранения/обновления: `Status 500`, тело ошибки пустое.
- В `POST /api/transform` для такого же сценария: `Status 400`, сообщение:
  - `Непредвиденная ошибка при применении трансформаций: An item with the same key has already been added. Key: x`
