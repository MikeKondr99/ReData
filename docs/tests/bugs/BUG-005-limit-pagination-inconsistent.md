# BUG-005: `limit` конфликтует с пагинацией (`total` не соответствует фактическим данным)

## Описание что не так
- После правок первая страница с большим `pageSize` начала возвращать корректное число строк по `limit`.
- Но осталась логическая несогласованность: при `total = 10` API на второй странице все еще возвращает 10 строк.
- По контракту `total` это "общее количество данных подходящих под составленный запрос", значит при `total=10` данных на странице 2 быть не должно.

## Шаги воспроизведения
- Выполнить `POST /api/transform` с телом:
  - `where true`
  - `limit 10 offset 0`
  - `pageNumber=1`, `pageSize=20`
- Получаем `total=10`, но `rows=20`.
- Повторить с `pageNumber=2`, `pageSize=10`.
- Получаем `total=10`, но снова `rows=10` (данные есть на второй странице).

## Ожидаемое поведение
- Если `limit=10`, то суммарно данных должно быть не больше 10.
- Для `pageNumber > 1` при `limit=10` ожидается пустая выдача (при `pageSize=10`).
- `total` и фактический объем данных должны быть согласованы.

## Возвращенный ответ API
- Ретест 1 (`where true + limit 10`, `pageNumber=1`, `pageSize=20`):
  - `Status 200`, `total=10`, `rows=10` (исправлено).
- Ретест 2 (`where true + limit 10`, `pageNumber=2`, `pageSize=10`):
  - `Status 200`, `total=10`, `rows=10`, ids `31..40` (несогласованность осталась).
- Ретест 3 (`groupBy + limit 2`, `pageNumber=1`, `pageSize=20`):
  - `Status 200`, `total=2`, `rows=2` (исправлено).
