# BUG-027: Потеря точности `num` при импорте/вычислениях (большие + высокоточные значения)

## Описание что не так
- Значения с высокой точностью теряют значимые разряды уже в данных набора, далее это видно и в `transform`, и в `export`.
- Примеры наблюдений:
  - `999999999999999999.123456789012345678` -> `1E+18`
  - `-999999999999999999.123456789012345677` -> `-1E+18`
  - `123456789.123456789012345678` -> `123456789.12345679`
- Из-за этого ломаются агрегаты:
  - для группы `A` ожидаемая сумма по исходным данным `2e-18`, фактически получается `1e-18`.

## Шаги воспроизведения
- Создать data connector с CSV, содержащим значения выше.
- Выполнить `POST /api/transform` c `select val` и `groupBy grp + SUM/AVG`.
- Сохранить аналогичный dataset и вызвать `export`.

## Ожидаемое поведение
- Сохранять и возвращать `num` без такой деградации точности (или явно валидировать/отклонять формат, если точность не поддерживается).

## Возвращенный ответ API
- Preview и export показывают уже усеченные значения:
  - `1E+18`, `-1E+18`, `123456789.12345679`.
- Групповые агрегаты считаются по этим усеченным числам.
