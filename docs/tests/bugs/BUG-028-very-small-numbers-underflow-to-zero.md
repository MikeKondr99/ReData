# BUG-028: Очень малые `num`-значения молча схлопываются в `0` (underflow)

## Описание что не так
- Сверхмалые значения (проверено на `1e-400` в десятичной записи) принимаются на импорте, но в данных становятся `0`.
- Потеря происходит без ошибки валидации/предупреждения.
- Дальше и preview, и export работают уже с нулями.

## Шаги воспроизведения
- Импортировать CSV, где `val = 0.` + `399` нулей + `1` (то есть `1e-400`) в двух строках.
- Выполнить `POST /api/transform` с `select val`.
- Выполнить `groupBy + SUM(val)`.
- Сохранить dataset и сделать `export`.

## Ожидаемое поведение
- Либо корректно хранить такие значения в `num`.
- Либо возвращать явную ошибку/валидацию, что точность/диапазон не поддерживается.
- Но не тихо заменять на `0`.

## Возвращенный ответ API
- `transform` (`select`): `val = 0` в обеих строках.
- `transform` (`SUM`): `sum_val = 0` при `cnt = 2`.
- `export`: в CSV также `0`.
