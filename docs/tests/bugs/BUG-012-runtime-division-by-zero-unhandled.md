# BUG-012: Runtime `division by zero` приводит к неуправляемому `500`

## Описание что не так
- Выражения с делением на ноль (`1 / 0`) проходят в сохраненный набор.
- На выполнении это приводит к `500`.
- Формат ошибки непоследователен:
  - в `select` возвращается структурированный `500` с текстом БД;
  - в `where` уходит stack trace приложения.

## Шаги воспроизведения
- `POST /api/transform` с `select: "1 / 0"` -> `500`.
- `POST /api/transform` с `where: "1 / 0 > 1"` -> `500` со stack trace.
- Создать набор с `select: "1 / 0"` -> `201`.
- Выполнить `export` этого набора -> `500`.

## Ожидаемое поведение
- Runtime-ошибки выражений должны возвращать единый безопасный формат ошибки без stack trace.
- Набор с заведомо аварийной трансформацией не должен бесшумно становиться неисполняемым.

## Возвращенный ответ API
- `select` вариант: `Status 500`, `22012: division by zero`.
- `where` вариант: `Status 500`, stack trace (`Pattern.Unions.Result.Unwrap`, `TransformEndpoint`).
- `export` набора: `Status 500`, тело пустое.
