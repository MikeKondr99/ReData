# Косяки OpenAPI и описания

## 1) Swagger JSON URL
- В UI фигурирует `"/swagger/v1/swagger.json"`, но рабочая схема доступна по `"/openapi/v1.json"`.
- Для автогенерации клиентов нужно брать `http://localhost:5223/openapi/v1.json`.

## 2) `restOptions` тип в `select`
- В схеме `SelectTransformation.restOptions` задан как enum integer.
- В фактическом `GET /api/datasets/{id}` приходит строка (`"NoAction"` / `"Delete"`).

## 3) `GET /api/datasets/{id}/export` статус-код
- В OpenAPI указан `204 No Content`.
- По факту при успешной выгрузке приходит `200` и файл (`text/csv`, attachment).

## 4) `PUT /api/datasets/{id}` и `id` в body
- По OpenAPI `UpdateDataSetRequest` не требует `id` в body.
- Фактическое поведение endpoint зависит от `id` в body и его согласованности с path.

## 5) `additionalProperties: false` не соблюдается
- В нескольких моделях схема запрещает дополнительные поля.
- Фактически API принимает extra-поля и молча их игнорирует (`POST/PUT /api/datasets`, `POST /api/transform`).
## 6) Код ответа на конфликт имени набора
- В OpenAPI для `POST /api/datasets` описан `409`.
- По факту дубль имени возвращает `400`.
## 7) `export` для отсутствующего набора
- Фактически endpoint возвращает `404`.
- `404` не описан в OpenAPI для `GET /api/datasets/{id}/export`.

## 8) Ограничения пагинации в `POST /api/transform`
- По факту `pageNumber` валидируется как `>= 1`, `pageSize` валидируется как `<= 100`.
- В схеме OpenAPI для `TransformRequest.pageNumber/pageSize` не заданы явные `minimum/maximum`, из-за чего контракт не отражает реальные ограничения.
